# Thermal simulation, 3D cable routing, and energy monitoring modules

Three calculation modules added to the single-file HTML app covering cable thermal behavior, containment fill rate, and energy cost/power quality analysis. All three follow the established pattern: standard-referenced lookup tables, conservative rounding (ceil for temperatures/fill/THD/capacitor sizing, floor for power factor), click-only UI with range sliders and button selectors, trilingual support via T.da/T.en/T.fa plus the _FA fallback dictionary. The derating table values align with DS/HD 60364-5-52 Tables B.52.14/15 and B.52.17. Fifty new tests bring the total to 160, all passing.

**Watch for:** The `renderThermal` derivation of Iz from `In * 1.45` conflates the MCB conventional trip factor with the cable's current-carrying capacity, which are not the same thing (confirmed). `thermalCalcDerating` silently reads `thermalState.selectedCableType` from global state rather than accepting it as a parameter (confirmed). The PQ traffic-light section in the energy module hard-codes measured values (230V, 50Hz, Plt=0.5, 1.0% unbalance) instead of deriving them from the load model, making it purely decorative (confirmed).

## High-level view

The thermal module implements the DS/HD 60364-5-52 temperature-rise formula correctly, but the main engineering concern is how "Iz" (the cable's current-carrying capacity) is derived — it's computed as `In * 1.45` inside `renderThermal`, treating the MCB conventional trip multiple as if it defines the cable's ampacity. Real cable sizing selects Iz from Table B.52.2 through B.52.12 based on cross-section, not from the breaker rating. This means the thermal display can show "SAFE" for a cable that is actually undersized.

The 3D cable routing module provides fill-rate calculation against containment cross-section area, grouping derating lookup, minimum bend radius checking (6x for PVC, 8x for XLPE), and a heat dissipation estimate. However, it allows up to 12 cables in a tray while clamping the derating lookup at 9, which is non-conservative for counts above 9.

The energy monitoring module implements reactive power compensation sizing, THD aggregation with a 0.7 diversity factor, and hourly tariff costing against 2024 Energinet rates. The power quality traffic light is mostly static — only THD varies with user input; the other four indicators always display green because their measured values are hardcoded constants.

## Iz derivation in renderThermal conflates breaker rating with cable capacity

In `renderThermal`, the cable's current-carrying capacity is computed as:

```javascript
var IzBase = st.ratedCurrent * 1.45; // MCB must trip at 1.45*In per IEC 60898
```

The 1.45 factor is the conventional tripping current of an MCB (IEC 60898-1 clause 8.6.2.2) — it defines the current at which the breaker *must* disconnect, not the continuous current the cable can carry. The correct Iz comes from DS/HD 60364-5-52 Tables B.52.2 through B.52.12, indexed by cross-section area and installation method. Using `In * 1.45` means that a 32A MCB protecting a 6mm² cable (Iz ~ 41A on tray) and a 32A MCB protecting a 10mm² cable (Iz ~ 57A on tray) would show identical thermal results, which is physically incorrect.

For a dimensioning tool referenced in life-safety work, this matters: the user could see "SAFE" for an undersized cable because Iz was artificially inflated by the 1.45 factor. **Confidence: confirmed** — visible at line ~9303 in `renderThermal`.

This is the only calculation function in the three modules with this pattern; `thermalCalcTemp` correctly accepts `cableType` as a parameter. The tests work around it by mutating `thermalState.selectedCableType` before calling the function. **Confidence: confirmed.**

## PQ traffic light with hardcoded measurements

In `renderEnergy`, the power quality status is computed with:

```javascript
var pqResult = energyGetPQStatus(230, 50.0, thdResult.thd, 0.5, 1.0);
```

Only the THD parameter varies based on user input. Voltage is always 230V (always "good"), frequency always 50.0Hz (always "good"), flicker always 0.5 (always "good"), unbalance always 1.0% (always "good"). The traffic light will always show green for 4 out of 5 indicators regardless of the load scenario. A user might conclude their installation has no voltage or unbalance concerns when those parameters aren't being evaluated. **Confidence: confirmed.**

## Grouping beyond table range

The cable3d module allows up to 12 cables in a tray (`if (cable3dState.cables.length < 12)`) but the derating lookup clamps at 9:

```javascript
var gc = Math.min(Math.max(groupCount, 1), 9);
```

DS/HD 60364-5-52 Table B.52.17 includes notes for groups > 9 circuits indicating that further reduction should be applied. When the cable count exceeds 9, the derating applied is that of 9 cables (e.g., 0.70 for tray), which is non-conservative for the actual group size. The thermal module's UI caps at 9 buttons so it's not affected, but the cable3d module needs either a warning or a more conservative factor for counts above 9. **Confidence: confirmed.**

## Test coverage gaps

Not tested: behavior at boundary conditions (exactly at maxTemp, exactly at fill limit), interpolation between ambient temperature steps (e.g., 27°C which falls between table entries), thermal behavior when `loadFactor > 1.0` combined with high ambient and grouping simultaneously, the static PQ indicators, cable3d derating behavior when cable count exceeds 9, or cross-module consistency between the thermal module's derating output and the cable3d module's derating output for the same configuration.
