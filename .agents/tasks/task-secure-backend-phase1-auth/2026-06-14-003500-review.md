# Auth, RBAC, and Case CRUD for multi-tenant finance backend

This change introduces the authentication layer (Argon2id password hashing, server-side sessions with hashed tokens, TOTP MFA), a role-based access control middleware, and tenant-scoped Case CRUD routes. It also brings the Prisma migration creating all database tables, integration tests against a real Postgres instance, and a CI workflow update with a Postgres service container. The implementation follows a conservative security posture appropriate for a finance application: passwords are never stored plaintext, session tokens are SHA-256 hashed before storage, cookies are httpOnly/secure/sameSite=strict, and error messages are generic to prevent user enumeration.

**Watch for:** Cross-tenant case creation via arbitrary `projectId` (confirmed), MFA secret stored in plaintext in the database (confirmed), no session invalidation on password or MFA change (confirmed), unhandled Prisma constraint errors on register (confirmed).

## High-level view

The session system uses 32 random bytes with only the SHA-256 hash stored in the database, proper expiry and revocation checks, and correct cookie flags. The weakness is that credential changes (password reset, MFA enrollment) don't invalidate existing sessions, so a compromised session outlives a password change.

Tenant isolation in Case CRUD relies on injecting `tenantId` into every Prisma WHERE clause. The GET, PATCH, and DELETE paths all enforce this correctly. The critical gap is in POST /cases: the route accepts an arbitrary `projectId` from the request body and creates a Case with `tenantId` from the session, but never verifies that the given `projectId` actually belongs to that tenant. An authenticated user in tenant A can create cases linked to a project in tenant B.

The MFA implementation stores `mfaSecret` as a plaintext column in the User table. For a finance app, this secret should be encrypted at rest using the `DATA_ENCRYPTION_KEY` already defined in the config schema. Anyone with database read access (backup leaks, SQL injection in a future feature) can generate valid TOTP codes for any user.

1. **Cross-tenant case creation via unvalidated projectId** — POST /cases accepts any `projectId` without verifying it belongs to the authenticated user's tenant. Add a check that the project exists and its `tenantId` matches `request.tenantId` before creating the case.
2. **MFA secret stored in plaintext** — `mfaSecret` is a raw base32 string in the User table. Encrypt it with `DATA_ENCRYPTION_KEY` before storage and decrypt on read during TOTP verification.
3. **No session invalidation on credential change** — MFA enrollment/verification doesn't revoke existing sessions. A stolen session continues working after MFA is enabled. After any credential change, revoke all sessions for that user except the current one.
4. **Register endpoint lacks error handling** — The `$transaction` doesn't catch Prisma errors. A constraint violation (P2002) or timeout propagates as a 500 with potentially sensitive stack trace details. Catch and return a generic 409/400.
5. **Expired sessions accumulate without cleanup** — Sessions are checked for expiry on validation but never deleted. Over time, the Session table grows unbounded. Add a periodic cleanup job or TTL-based pruning.

## Cross-tenant case creation via unvalidated projectId

The POST /cases handler in `src/routes/cases.ts` receives `projectId` from the request body, validates its format with zod (must be a non-empty string), then passes it directly to `prisma.case.create` alongside `request.tenantId`:

```typescript
const created = await prisma.case.create({
  data: {
    tenantId: request.tenantId,
    projectId: data.projectId,  // never verified against tenant
    problem: data.problem,
    ...
  },
});
```

The Case table has a foreign key to Project (`Case_projectId_fkey`), but no foreign key from `Case.tenantId` to Tenant. Postgres will only verify the project *exists*, not that it belongs to the same tenant. An authenticated user can supply any valid `projectId` — including one belonging to another tenant — and successfully create a case referencing that project. The case will have tenantId=A but projectId belonging to tenant B, creating an inconsistency that could expose cross-tenant information depending on how downstream features query project data.

Fix: before creating the case, verify `await prisma.project.findFirst({ where: { id: data.projectId, tenantId: request.tenantId } })` returns a result. Return 404 otherwise.

## MFA secret at rest

The `mfaSecret` column on User stores raw TOTP secrets as base32 strings. The `DATA_ENCRYPTION_KEY` config field exists in the schema specifically for encrypting sensitive data at rest, but it's not used anywhere in this change. A database breach that exposes `mfaSecret` values effectively disables MFA for all affected users — an attacker who already has password hashes can now also generate valid TOTP codes.

## Session lifecycle gaps

When MFA is enrolled or verified (`POST /auth/mfa/verify`), the user's existing sessions remain valid. If a session is stolen before MFA is enabled, it continues working after MFA is enabled on the account — MFA only gates new logins, not existing sessions. The same will apply when password-change is implemented: without revoking all existing sessions, a compromised session survives a password reset.

The `revokeSession` function operates on a single tokenHash. There's no mechanism to revoke all sessions for a user. A "revoke all" utility (or a `generation` counter on the user row that invalidates all older sessions) is needed for credential-change flows.

Expired and revoked sessions are never deleted. The `Session_expiresAt_idx` index helps query performance, but the table will accumulate dead rows indefinitely in production.

## Registration error handling

The register handler wraps tenant+user+membership creation in `prisma.$transaction`, but doesn't catch the resulting error. If the transaction fails (unique constraint, connection timeout, deadlock), Fastify returns a 500 with potentially sensitive error details. The P2002 error in particular would leak index names and constraint details. The register endpoint should catch Prisma errors and return a generic 409 or 400, matching the defensive pattern already used in the login endpoint.

## Test coverage gaps

Not tested: rate-limiting behavior (tests bypass it via allowList), expired sessions being rejected, the complete MFA-required login flow (attempting login with password only when MFA is enabled, then retrying with mfaToken), the `requireRole` middleware at levels other than OWNER (all test users are OWNERs, so the MANAGER/ADMIN/VIEWER rejection paths are never exercised), and malformed/tampered cookie values.
