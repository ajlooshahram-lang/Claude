# QI Platform Phase 2: UI-API integration (v2)

The v1 review raised 6 issues, of which 4 required fixes. All 4 have been addressed: fire-and-forget API calls now have `.catch()` handlers that dispatch `qi-sync-error` and console-warn; the dead-code MFA branch and double `res.data &&` typo are gone; the network-fallback heuristic checks for auth-related keywords rather than exact string matches; and the MFA reveal logic is extracted into `revealMfaChallenge()`. All three test suites pass.

**Watch for:** sync-error event is dispatched but never surfaced to the user as a toast (confirmed); `checkSessionAndBoot` has seven near-identical boot-sequence blocks with non-trivial drift risk (confirmed).

**Verdict**: APPROVED

## High-level view

The sync-error notification path dispatches a `qi-sync-error` CustomEvent from `store.js`, but nothing in `ui.js` listens to it. Users without DevTools open will never learn that a mutation failed to sync. The event contract is in place; wiring a listener is a one-liner to close the gap.

The session-boot logic in `checkSessionAndBoot` and its surrounding init block repeats the same 4-line "resolve initial hash and call `go()`" pattern seven times. A single extracted helper would eliminate drift risk and reduce the function by ~40%. Readability concern, not a correctness bug.

The network-fallback heuristic now matches auth-keyword substrings. Any non-auth error triggers fallback to local mode — fail-offline rather than fail-blocked, which is the safe direction for the "never lose data" requirement.

## Sync-error event gap

The fix adds `notifySyncError()` in `store.js` which dispatches `qi-sync-error` on the window and logs to console:

```javascript
function notifySyncError(action, err) {
  var msg = "Sync failed (" + action + "): change saved locally";
  if (typeof root.dispatchEvent === "function") {
    root.dispatchEvent(new (root.CustomEvent || root.Event)("qi-sync-error", { detail: { action: action, error: err } }));
  }
  if (typeof console !== "undefined") console.warn("[QI sync]", msg, err);
}
```

The existing pattern for `qi-storage-error` in `ui.js` (line ~2147) shows exactly how to consume such events: `window.addEventListener("qi-storage-error", () => { toast(…) })`. The same one-liner for `qi-sync-error` would complete the user-facing feedback loop. Without it, the event fires into the void unless external code subscribes.

## Boot-sequence repetition

`checkSessionAndBoot` contains five exit paths, each computing:

```javascript
const initialHash = (location.hash || "").replace(/^#/, "");
go(initialHash && RENDER[initialHash] ? initialHash : "dashboard", { skipHash: !!(initialHash && RENDER[initialHash]) });
```

Two more identical blocks exist in the synchronous boot path at the end of the IIFE. A `bootDefault()` one-liner would replace all seven. If the hash-resolution logic changes (e.g., to support deep-links with query params), seven sites need updating.
