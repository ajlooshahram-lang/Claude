# Semantic Review (v1) - QI Platform Phase 2: UI-API Integration

**Verdict**: NEEDS_CHANGES

## Summary

The QI Platform gains an opt-in online mode: a thin API client (`api.js`) wraps all backend endpoints, the store proxies mutations through it when mode is set to `"api"`, and the UI presents a login/register gate before showing the dashboard. The existing offline flow (mode `"local"`) is the default and runs unchanged.

## Issues (6)

1. **Fire-and-forget sync loses mutations silently** - Store wraps case mutations with API calls inside `try/catch` that swallows errors. If the server rejects a create/update/delete, local state diverges with no user notification. The promise returned by QIAPI is neither awaited nor `.catch()`-ed. At minimum, `.catch()` the promise and show a toast ("Sync failed - change saved locally").

2. **MFA challenge on ok:true response** - `handleAuthSubmit` treats `res.ok && res.data.mfaRequired` as "login not complete, show MFA field." The error-branch duplicate (`res.data && res.data && res.data.mfaRequired`) has a typo (double `res.data &&`) and is dead code because error normalization sets `data: null`.

3. **Narrow network-failure heuristic** - `checkSessionAndBoot` only falls back to local mode when `res.error` is exactly `"fetch not available"` or `"Network error"`. A 500, CORS error, or browser-specific DNS failure message will show the login screen instead of falling back. A more robust check: treat any non-401/non-403 error from `me()` as "server unreachable" and fall back.

4. **Backend URL stored in two places** - `localStorage("qi_api_url")` and `window.QI_API_URL`. Minor concern but could desync.

5. **Connection dot is not periodically refreshed** - `connStatus` is set at boot/mode-switch but never polled afterward. Acceptable for v1 but worth noting.

6. **Duplicate MFA handling code** - The ok and error branches in `handleAuthSubmit` duplicate the same MFA-reveal logic (identical ~6 lines). Extract into a helper.

## Required Fixes (items 1, 2, 3, 6)

- **Issue 1**: Add `.catch()` to the fire-and-forget API calls in store.js that dispatches a `qi-sync-error` event or logs a warning. This ensures users know when server sync fails.
- **Issue 2**: Remove the dead-code MFA branch in the error path of `handleAuthSubmit`. Fix the double `res.data &&` typo.
- **Issue 3**: Make the network-failure heuristic more robust: if `me()` returns a non-401/non-403 status (e.g., res.error does not contain "authenticated" or "authorized"), treat it as "server unreachable" and fall back to local mode.
- **Issue 6**: Extract MFA reveal logic into a small helper function.

## Acceptable as-is (items 4, 5)

- Issue 4: Dual storage is a minor concern, acceptable for v1.
- Issue 5: No periodic polling is fine for this iteration.
