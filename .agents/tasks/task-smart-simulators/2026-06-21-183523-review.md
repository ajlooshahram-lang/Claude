# Interactive SVG circuit simulators across 6 modules

Twelve pure functions produce inline SVG circuit diagrams that visualize current flow, impedance distribution, voltage drop, short-circuit fault paths, three-phase topology, and motor rotor motion. The diagrams are generated from the same calculated values already displayed numerically, so they stay in sync with user input without extra state. CSS animations (marching dashes, rotor spin) provide visual current flow that scales proportionally to the computed current magnitude via `animation-duration`.

Watch for: **gradient ID collision** in `renderVdropSim` if the function is ever called more than once per page render (confirmed); **`simHeatColor` called with Infinity** when `Ri` is zero in the EMF simulator (likely); **no guard against very large resistor arrays** in `renderDCSeriesParallelSim` causing SVG overflow (possible).

## High-level view

The implementation follows the app's existing pattern of pure render functions returning HTML strings, injected via `+=` into the parent module's output. Each simulator function takes the same numeric results already computed by the corresponding `calc` function, so no new state or recomputation is introduced. The visual encoding maps wire animation speed inversely to current, resistor fill color to dissipated power, and component widths to impedance share.

Reduced-motion is handled at the CSS layer with both a media query and the `body.motion-off` class. One gap: `.sim-resistor` and `.sim-source` `transition` properties are not disabled under reduced motion — these are `0.3s` fill color transitions that fire on re-render. (possible — depends on accessibility policy)

Test coverage is solid for return-value contracts and empty-input guards, but no tests exercise boundary values (`Ri=0`, extremely large arrays, `n=1` rpm) where the visual output degrades silently.

<details><summary>Issues (5)</summary>

1. **Gradient ID collision in `renderVdropSim`** — The hardcoded `id="vdGrad"` will produce duplicate IDs if this SVG is ever rendered more than once per document state. Currently safe (single module visible), but fragile. Use a unique suffix or inline gradient style instead. (confirmed)
2. **`simHeatColor` receives `E*E/Ri` when Ri=0** — `renderDCEmfSim` passes `simHeatColor(Ploss, E * E / Ri)` as Pmax. When `Ri=0`, Pmax is `Infinity`, and the guard (`!Pmax || Pmax <= 0`) doesn't catch it. Result: ratio is always 0, coloring is always cold regardless of actual power — a silent visual bug. Add `Ri || 1` or clamp Pmax to a finite ceiling. (likely)
3. **No resistor-count cap in `renderDCSeriesParallelSim`** — Parallel mode height is `40 + n * 35`. At 20+ resistors the SVG becomes 740px+ with no truncation. Currently gated by UI presets, but the function itself has no guard. (possible)
4. **Rotor `animation-duration` extremely large at low speeds** — `60 / Math.max(1, n)` at 1 rpm yields a 60-second period, visually indistinguishable from stopped but still burning compositor cycles. Consider disabling animation below ~10 rpm. (likely)
5. **`.sim-resistor` / `.sim-source` transitions not disabled under reduced motion** — These 0.3s fill transitions are not covered by the `prefers-reduced-motion` or `motion-off` rules. Color-only transitions are arguably acceptable, but strict accessibility guidelines may require instant transitions. (possible)

## `simHeatColor` Infinity path in EMF simulator

`renderDCEmfSim` at line 26137 passes `E * E / Ri` as the `Pmax` argument to `simHeatColor`. The guard inside `simHeatColor` is `if (!Pmax || Pmax <= 0) Pmax = 1000` — this does not catch `Infinity` because `Infinity > 0`. The resulting ratio `P / Infinity` evaluates to `0`, producing a permanently cold-blue fill on the internal resistance block regardless of actual dissipation. The fix is a one-line clamp: `if (!Pmax || Pmax <= 0 || !isFinite(Pmax)) Pmax = 1000`.

## Series/parallel SVG scaling

In series mode, `segW = Math.min(50, (w - 80) / n)` keeps segments within the 300px viewBox even at high `n`, but labels shrink to illegibility below ~8px width. In parallel mode, height grows linearly without bound. The existing UI constrains this through button presets (typically 2-5 resistors), so this is a latent issue rather than a current bug. A defensive cap of `Math.min(n, 10)` with an ellipsis indicator would future-proof it.

## Motor rotor period at extreme values

The motor simulator correctly maps synchronous speed to animation period (`60 / n` seconds = one visual revolution per actual revolution). At standard motor speeds (750-3000 rpm) this produces 0.02-0.08s periods — fast and visually effective. The edge case is low-speed applications or startup scenarios where `n` approaches single digits. The animation technically runs but appears frozen, which could confuse users into thinking the visualization is broken.

## Test coverage gaps

Not tested: `Ri=0` input to `renderDCEmfSim` (triggers the Infinity path), resistor arrays with >10 entries, `n=1` motor speed, negative current values passed to `simFlowDuration`, and dark-mode theme variable resolution in SVG text fills.
