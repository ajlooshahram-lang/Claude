# Multi-tenant CRUD, share tokens, MFA encryption, and deployment packaging

Phase 3 adds the remaining domain CRUD endpoints (projects, registers, snapshots), a token-based project sharing system, MFA-secret-at-rest encryption, a password change flow with session revocation, and Docker-based deployment packaging. The server grows from a minimal auth+cases skeleton to a deployable multi-tenant platform with 83 passing integration tests. The frontend gains an API client module and fire-and-forget sync wrappers that mirror local mutations to the backend.

Watch for:
- Share token scope field is stored but never enforced — MANAGER tokens grant no write access, and VIEWER tokens return the full dataset regardless (confirmed).
- AES-256-GCM key derivation truncates/pads a UTF-8 string to 32 bytes without a proper KDF; short or non-ASCII keys produce a weak effective key (confirmed).
- Snapshot restore performs sequential row-by-row INSERTs without a transaction; a failure partway through leaves the project in an inconsistent state (confirmed).
- The frontend sync pattern is fire-and-forget without server-ID reconciliation — local IDs diverge from server IDs permanently (confirmed).

## High-level view

The four new CRUD modules follow the pattern established by `cases.ts`: Fastify plugin, Zod validation, Prisma queries scoped by `tenantId`, soft-delete via `deletedAt`. Tenant isolation is enforced at the query level rather than at middleware, so every route must include the `tenantId` filter — but cross-tenant enumeration returns 404 rather than 403, a deliberate privacy choice.

The share-token system stores only the SHA-256 hash of the token in the database. The public `/shared/:token` endpoint bypasses authentication. However, the `scope` field (VIEWER/MANAGER) is persisted but never enforced — both token types serve identical data.

MFA secrets are encrypted with AES-256-GCM, but the key is consumed as raw UTF-8 bytes truncated or zero-padded to 32 rather than derived through a proper KDF. Production config requires the key but doesn't enforce its entropy.

The deployment stack uses docker-compose with network isolation between the database and the nginx frontend. Postgres credentials are hardcoded in the compose file; the actual secrets (SESSION_SECRET, DATA_ENCRYPTION_KEY) use shell substitution but have no `.env.example` documenting requirements.

The frontend API-sync layer wraps local-first store mutations with fire-and-forget API calls. Failures dispatch custom events but are never retried. Local and server IDs permanently diverge because the store generates client-side IDs that the server never sees.

<summary>Issues (7)</summary>

1. **Share token scope is decorative** — The `scope` field (VIEWER/MANAGER) is persisted and returned but never used to filter data or gate write access on the public endpoint. Either implement scope-based filtering or document it as a future feature so it's not mistaken for a security boundary.
2. **Weak key derivation for MFA encryption** — `DATA_ENCRYPTION_KEY` is consumed as raw UTF-8 bytes, truncated to 32 or zero-padded if shorter. Use HKDF or at minimum validate the key is exactly 32 hex-encoded bytes to avoid weak effective keys.
3. **Snapshot restore is not transactional** — The restore endpoint deletes existing cases/registers then creates new ones row-by-row without wrapping the operation in `prisma.$transaction`. A crash or constraint violation mid-restore leaves the project empty.
4. **ID divergence between frontend and backend** — Local store generates IDs client-side (`uid()`), then the API sync creates a separate record with a server-generated `cuid()`. No reconciliation maps the two, so subsequent updates/deletes on the local ID won't match the server record.
5. **Hardcoded DB credentials in docker-compose** — `POSTGRES_USER: qi` and `POSTGRES_PASSWORD: qi` are committed in plaintext. Add a `.env.example` documenting all required environment variables.
6. **Snapshot restore fragile to schema evolution** — The `...rest` spread + `as never` cast on restored records means any extra fields in old snapshots (from schema changes) will cause a runtime Prisma rejection with no compile-time safety net.
7. **Session cleanup deletes revoked sessions immediately** — `cleanExpiredSessions` hard-deletes revoked sessions. If a security audit requires session history (who was logged in when), this data is lost 15 minutes after revocation.

## Share token scope is stored but not enforced

The public `GET /shared/:token` endpoint returns `scope` in the response payload, but the data served is identical regardless of the scope value — the full project, all cases, and all registers are returned for both VIEWER and MANAGER tokens:

```typescript
// shares.ts, /shared/:token handler — no scope branching
return {
  project,
  cases,
  registers,
  scope: shareToken.scope, // returned but unused
};
```

A consumer could reasonably assume that a VIEWER token won't expose sensitive fields or that a MANAGER token grants write access via some endpoint. Neither is true. The field's existence implies access control that doesn't exist.

## MFA encryption key handling

The `crypto.ts` encrypt/decrypt functions accept a key as a UTF-8 string, slice it to 32 bytes with `Buffer.from(key, "utf8").subarray(0, 32)`, and zero-pad if shorter:

```typescript
const keyBuffer = Buffer.from(key, "utf8").subarray(0, 32);
if (keyBuffer.length < 32) {
  const padded = Buffer.alloc(32); // zero-filled
  keyBuffer.copy(padded);
  // ...use padded as key
}
```

Two problems: a multi-byte UTF-8 character can cause the effective key to be shorter than expected in bytes after the subarray; and zero-padding a short key dramatically reduces entropy — a 16-character ASCII key becomes a 32-byte key where half the bytes are null. The config schema only enforces `z.string().min(16)` (16 characters, not bytes). A proper approach would accept a 64-character hex-encoded key or derive the encryption key via HKDF.

## Snapshot restore lacks transactional safety

The restore endpoint performs: (1) create a pre-restore backup snapshot, (2) `deleteMany` existing cases, (3) `deleteMany` existing registers, (4) loop through snapshot data creating new cases one by one, (5) loop through snapshot data creating registers one by one. None of steps 2-5 are wrapped in a `prisma.$transaction`. If the process crashes after step 2 but before step 4 completes, the project has lost its current data and only partially restored from the snapshot.

The sequential `for...of` + `await prisma.case.create` pattern also means N+1 round trips to the database for N records. `createMany` would be both faster and easier to wrap in a transaction.

## Frontend fire-and-forget sync with permanent ID mismatch

The store.js sync wrappers call the local mutation first (which generates a client-side ID via `uid()`), then fire the API call. The API call creates a *new* server-side record with a `cuid()` ID:

```javascript
addProject = function (name) {
  var result = _addProject(name);  // local ID: "c1n5k7x3ab2f"
  // API creates a new record with server-generated cuid: "clx9abc..."
  QIAPI.createProject({ name: name || "New Project" });
  return result;
};
```

When the user later calls `deleteProject(localId)`, the API fires `QIAPI.deleteProject(localId)` which will 404 on the server. This pattern works only if the server state is treated as a write-only backup that's never read back; it breaks for bidirectional sync or multi-device scenarios.

## Docker deployment concerns

The nginx configuration's `Content-Security-Policy` allows `'unsafe-inline'` for both scripts and styles, which weakens XSS protection. The api service runs `npx prisma migrate deploy` on every container start — correct for single-instance but will race if scaled to multiple replicas.

## Snapshot restore fragility to schema evolution

During restore, the code destructures each case/register from the snapshot JSON:

```typescript
const { id: _id, createdAt: _ca, updatedAt: _ua, deletedAt: _da, ...rest } = c;
await prisma.case.create({
  data: { ...rest, tenantId: request.tenantId, projectId } as never,
});
```

The `as never` cast suppresses all type checking. If the snapshot was taken under an older schema (with columns that no longer exist, or missing required new columns), Prisma will throw at runtime. Since snapshots are long-lived data, schema migration strategies for snapshot payloads should be considered.
