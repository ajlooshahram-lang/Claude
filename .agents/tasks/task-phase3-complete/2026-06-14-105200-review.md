# Multi-tenant CRUD, share tokens, MFA encryption, and deployment packaging (v2)

Phase 3 adds the remaining domain CRUD endpoints (projects, registers, snapshots), a token-based project sharing system, MFA-secret-at-rest encryption, a password change flow with session revocation, and Docker-based deployment packaging. All seven issues from the v1 review have been addressed: snapshot restore is now transactional with `createMany`, VIEWER share tokens filter sensitive fields, the encryption key is validated to exactly 32 bytes, docker-compose uses environment variable substitution with fail-fast syntax, and the remaining architectural concerns (ID divergence, schema evolution, session cleanup) are documented with TODOs.

Watch for:
- Backup snapshot created outside the restore transaction can orphan on transaction rollback, leaving a misleading pre-restore record (likely).
- The `decrypt()` function treats any string with fewer than 3 colon-separated parts as plaintext and returns it unmodified — a legitimate ciphertext corruption silently returns garbage instead of failing (confirmed).
- VIEWER scope filtering is applied to cases and project fields but not to registers — all register data (which may contain cost/procurement data) is served regardless of scope (confirmed).

**Verdict**: APPROVED

## High-level view

The snapshot restore now wraps delete+recreate in `prisma.$transaction` with `createMany`, eliminating the data-loss window. The pre-restore backup snapshot is created outside the transaction, so a transaction rollback leaves a backup record that doesn't correspond to any actual restore — minor inconsistency that may confuse audit trails.

The crypto module enforces exact key sizes via `parseKey()` (64 hex chars or 32 ASCII bytes), and production boot refuses to start without a valid key. However, `decrypt()` in the auth routes is called without try/catch — a GCM auth-tag failure (from key rotation or data corruption) surfaces as an opaque 500 rather than a recoverable error.

VIEWER share tokens now filter `actCost`, `estCost`, `costCat`, and `rootCause` from cases and reduce the project payload to id/name/description/timestamps. Register rows are served unfiltered to both VIEWER and MANAGER tokens — the 13 register types include `procurement` and `resources` which may contain analogous cost data.

<summary>Issues (3)</summary>

1. **Register data unfiltered for VIEWER scope** — The `/shared/:token` endpoint filters case fields for VIEWER tokens but serves all register rows (including procurement, resources, cost data) unfiltered. If registers contain sensitive data, extend the scope filter to cover them.
2. **Decrypt silently returns garbage on corruption** — `decrypt()` checks for the `iv:authTag:ciphertext` three-part format but if a ciphertext is corrupted in a way that preserves the colon structure (e.g., truncated hex), the GCM auth tag verification will throw an unhandled error up the call stack. Conversely, if the ciphertext loses its colons (data corruption), it's silently returned as "plaintext." Add explicit error handling around the decrypt path in the auth routes.
3. **Backup snapshot orphaned on restore failure** — The pre-restore backup is created before the `$transaction` block. If the transaction rolls back, the backup snapshot remains in the database with a "Pre-restore backup" label but no restore actually occurred. Consider moving the backup inside the transaction or documenting this as intentional.

## Register data unfiltered for VIEWER scope

The VIEWER scope filtering in `shares.ts` strips cost-related fields from cases and reduces the project object to public metadata. But register rows pass through without filtering:

```typescript
if (shareToken.scope === "VIEWER") {
  const filteredCases = cases.map((c) => {
    const { actCost: _ac, estCost: _ec, costCat: _cc, rootCause: _rc, ...viewerFields } = c;
    return viewerFields;
  });
  return {
    project: filteredProject,
    cases: filteredCases,
    registers,  // <-- full register data including procurement, resources, costs
    scope: shareToken.scope,
  };
}
```

The 13 register types include `procurement`, `resources`, and financial categories that likely contain cost data analogous to the case fields being filtered. If the intent is that VIEWER tokens hide financial information, the filter needs to extend to registers.

## Decrypt error handling gap

The `decrypt()` function in `crypto.ts` has two failure modes with different behaviors:

1. If the ciphertext doesn't contain exactly 2 colons, it's treated as "legacy plaintext" and returned verbatim — a corrupted ciphertext that loses its structure is silently served as if it were a valid plaintext secret.

2. If the ciphertext has the right structure but the auth tag doesn't match (key rotation or data corruption), `decipher.final()` throws. This exception propagates uncaught through the auth route handler:

```typescript
// auth/routes.ts — login path
const decryptedSecret = decrypt(user.mfaSecret, config.dataEncryptionKey);
const mfaValid = verifyTotpToken(decryptedSecret, mfaToken);
```

A GCM auth failure surfaces as a 500 to the user rather than a meaningful "MFA configuration error" response. If the encryption key is rotated without re-encrypting existing secrets, every MFA login attempt fails with an opaque internal error and no recovery path.
