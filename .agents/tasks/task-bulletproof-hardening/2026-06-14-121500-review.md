# Bulletproof hardening layer for QI Platform

This change introduces a comprehensive security middleware stack (request-id, CSRF double-submit, body limits, error sanitization, security headers), centralized Zod strict-mode validation, structured audit logging, soft-delete enforcement, a metrics endpoint, graceful shutdown, and 17 dedicated security hardening tests plus SQL injection static analysis. The approach layers defenses from outside-in: transport-level protections first (headers, limits), then protocol-level (CSRF), then schema-level (Zod strict), then data-level (soft-delete middleware, parameterized queries).

Watch for: **`validateId` reply pattern** that fires `void reply.send()` then returns bare `undefined` from async handlers, which the FEAT-004 findings acknowledge causes "Cannot write headers" conflicts with onSend hooks (confirmed). **CSRF token never rotates** after initial login — the same token persists for the lifetime of the session (confirmed). **Metrics endpoint accessible via `X-Forwarded-For` spoofing** because `trustProxy: true` lets any client claim to be `127.0.0.1` (confirmed). **Audit log silent failures** are invisible — no monitoring path exists for audit write failures (likely).

## High-level view

The CSRF implementation uses double-submit cookies: the server sets a non-httpOnly `csrf_token` cookie on successful auth, and requires the client to echo it back via `X-CSRF-Token` header on mutations. The token is generated once at login and never rotated, which weakens its protection window against cookie-exfiltration attacks.

The metrics endpoint relies on `request.ip === "127.0.0.1"` for access control, but `trustProxy: true` means this IP comes from `X-Forwarded-For`, making it spoofable by any client without a `METRICS_TOKEN` env var set.

Audit logs are fire-and-forget (`.catch(() => {})`) which means the system cannot detect if the audit trail has silently stopped writing — an operational gap for a security-critical feature.

The `validateId` utility produces a race condition between its `void reply.send()` call and Fastify's onSend hook chain (Permissions-Policy header), resulting in "Cannot write headers after they are sent" errors on any request with an invalid path parameter ID.

1. **`validateId` races with onSend hooks** — `void reply.code(400).send()` fires the response but the handler returns `undefined` (not the reply), so Fastify's onSend hooks (Permissions-Policy header) attempt to write headers on an already-sent response. Return `reply` from the handler when `validateId` returns false.
2. **CSRF token never rotates** — A token issued at login persists indefinitely. If the cookie is exfiltrated (e.g., via a subdomain XSS), the attacker can forge requests for the lifetime of the session. Rotate the CSRF token on sensitive operations or on every response.
3. **Metrics endpoint bypassable via `X-Forwarded-For`** — With `trustProxy: true`, any client can set `X-Forwarded-For: 127.0.0.1` and pass the localhost check. Require `METRICS_TOKEN` in production or check `request.socket.remoteAddress` directly.
4. **Audit log failures are silent** — `.catch(() => {})` on every audit write means a Prisma connection failure, schema drift, or disk-full scenario produces no alert. Add at minimum a counter or log-level warning inside the catch.
5. **Backup service sleeps before first dump** — The backup container runs `sleep 86400` before its first `pg_dump`, meaning a fresh deployment has no backup for the first 24 hours. Move the dump before the sleep.
6. **`responseTimes.shift()` is O(n) on a 10k-element array** — Called on every request once the array is full. Use a circular buffer index instead.
7. **Snapshot DELETE is a hard delete** — While all other entities (Project, Case, RegisterRow) use soft-delete, snapshots use `prisma.snapshot.delete()`. This is inconsistent and makes snapshot deletions irrecoverable and un-auditable post-facto.

## `validateId` reply ownership and Fastify lifecycle conflict

The `validateId` utility sends a 400 response via `void reply.code(400).send()` and returns `false`. All callers then do `if (!validateId(id, reply)) return;` — returning `undefined` from the async handler. In Fastify, when an async handler returns `undefined`, the framework assumes the handler hasn't finished with the reply. Meanwhile, the onSend hooks (specifically the Permissions-Policy header hook) fire and try to set headers on the response that `validateId` already sent.

The FEAT-004 findings explicitly note this caused test failures: "Cannot write headers after they are sent." The workaround was to avoid testing paths through `validateId` via path params. But this means the validateId path is producing unhandled rejections in production for any invalid ID in a URL parameter.

The fix is straightforward — callers should `return reply` (not bare `return`) and `validateId` should return the reply object:

```typescript
// Instead of:
if (!validateId(id, reply)) return;

// Should be:
if (!validateId(id, reply)) return reply;
```

## CSRF token lifecycle and rotation gap

The CSRF token is issued exactly once: on successful login or register. It is set as a cookie with `SameSite=Strict`, `Path=/`, and no expiration (session cookie). The token is never rotated — not on password change, not on privilege escalation, not per-request.

If the token is exfiltrated (via subdomain XSS, cookie-jar sharing, or browser extension), the attacker can use it for CSRF indefinitely. Rotating the token on sensitive operations (password change, MFA verify) or on every mutating response would limit the window. The `SameSite=Strict` attribute mitigates typical CSRF vectors, but the static token represents an elevated risk in session-hijacking scenarios where the attacker already has a cookie read primitive.

## Metrics endpoint trust boundary with `trustProxy: true`

```typescript
// metrics.ts
const ip = request.ip;
if (ip === "127.0.0.1" || ip === "::1" || ip === "localhost") {
  return true;
}
```

With `trustProxy: true` set globally in Fastify options, `request.ip` reflects the `X-Forwarded-For` header value, not the actual socket peer address. Any external client can send `X-Forwarded-For: 127.0.0.1` and gain access to the metrics endpoint, which exposes active session counts, response time distributions, error rates, and a database connectivity signal.

The `METRICS_TOKEN` fallback exists but is only checked when `process.env["METRICS_TOKEN"]` is set. In deployments where operators forget to set this env var, the endpoint is open to anyone who spoofs the header. The fix is to check `request.socket.remoteAddress` for the localhost case rather than the proxied `request.ip`.

## Audit log observability gap

Every mutation writes an audit log entry using the pattern:

```typescript
prisma.auditLog.create({ data: { ... } }).catch(() => { /* non-blocking */ });
```

If the AuditLog table becomes unavailable (migration drift, connection exhaustion, disk full), the system silently drops audit entries without any signal. For a security audit trail, this matters — you'd want at minimum a `logger.warn()` inside the catch, or a counter that the metrics endpoint can report.

## Backup service first-dump timing

The backup service's shell loop structure is:

```sh
while true; do
  sleep 86400
  # ... pg_dump ...
done
```

The first action is `sleep 86400`, meaning no backup is taken for the first 24 hours after container start. A fresh deployment or a container restart leaves the system with no recent backup. Moving the dump before the sleep (`do ... pg_dump ... sleep 86400 ... done`) would ensure immediate backup on start.

## Security hardening test coverage gaps

Not tested: the graceful shutdown path (503 from /health during shutdown), the Permissions-Policy header presence, the HSTS header value, or any scenario where multiple onSend hooks race on the same response (the validateId problem). The rate-limit test builds a separate app instance with `NODE_ENV=development` to avoid the test-mode allowlist, but this means the actual production rate-limit behavior (no allowlist at all) isn't directly validated.

## Soft-delete inconsistency for snapshots

The soft-delete middleware covers `Project`, `Case`, `RegisterRow`, `User`, and `Tenant`. Snapshots are excluded and use `prisma.snapshot.delete()` (hard delete). Deleted snapshots cannot be recovered, and if accidentally deleted, the point-in-time project state is lost permanently. This creates an asymmetry where the audit log records a "snapshot.delete" event but the data it references no longer exists.
